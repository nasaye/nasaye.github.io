### 에러 리포트

```
running wslexec: The service cannot be started, either because it is disabled or because it has no enabled devices associated with it. Wsl/0x80070422: c:\windows\system32\wsl.exe --terminate docker-desktop: exit status 0xffffffff (stderr: , wslErrorCode: Wsl/0x80070422)

```
---

### 상황

 깃 페이지에서 테마를 관리하기 위해 도커 데스크탑을 다운로드. PC 재부팅과 함께 에러메세지 발생. 


### 초기 대응

우선, wsl 이 작동하지 않아 발생한 오류이기 때문에, 해당 위치에 wsl이 있는지부터 확인한다.
   - fzf 로 확인 결과 해당 위치에 wsl.exe 가 존재함. 

 wslErrorCode 를 확인한다.
   - WSL Error Code 0x80070422은 주로 LxssManager(WSL 서비스)가 비활성화되어 있거나, 관련 네트워크/업데이트 서비스가 중단되어 발생합니다. 서비스 앱에서 'LxssManager'를 '자동'으로 변경하고, Windows 업데이트 서비스를 다시 활성화하여 해결할 수 있습니다. 
   - 
   - 윈도우 검색 결과, LxssMaaanager를 자동으로 변경해야 함. 서비스 창에 들어가 보니, LxssManager 이 정말 수동으로 되어 있었다. 하지만 자동으로 변경하고 적용을 누르자 서비스가 거부되었다는 메세지만 떴다.

서비스를 수동으로 시작하고, 도커를 다시 실행해 봤다. 
로딩이 끝나지 않는다. 

CLI 에 wsl -v, wsl --status 로 상태를 확인해 봤다.
wsl.exe는 설치되어 있었지만 status 에서 에러 발생.
`
 wsl --status
서비스를 사용할 수 없거나 서비스와 연관되어 사용 가능한 장치가 없기 때문에 서비스를 시작할 수 없습니다.
오류 코드: Wsl/0x80070422
`

수동으로 시작했지만 여전히 같은 오류가 발생함.

### 사례 검색

바로 GPT 서치를 하고 싶었지만, 너무 의존하게 될 것 같아 5분만 서칭을 해 봤다.
생각보다 골치 아픈 문제로 보였다. 
3년 전 레딧 포스팅에서 해결한 방법대로 wsl 중지-실행 등을 해보았지만 고쳐지지 않음.
검색 중 시간이 아까워서 GPT 사용


### GPT 사용

- 서비스 상태 확인

```PowerShell
sc qc LxssManager
sc qc vmcompute
sc qc hns
dism /online /get-featureinfo /featurename:Microsoft-Windows-Subsystem-Linux
dism /online /get-featureinfo /featurename:VirtualMachinePlatform

bcdedit /enum | findstr -i hypervisorlaunchtype
```

위 명령어를 통해 wsl 관련 서비스들이 정상 작동중인지 확인했고, 다 정상 작동중.

| Microsoft-Windows-Subsystem-Linux
| VirtualMachinePlatform

이 기능들을 통해 대략적으로 wsl 감 잡히는중.


- wsl2 에 이상이 있는지 의심

GPT와 체크해본 결과 구성요소들이 잘 설치되고 작동중이었기에, 기본 wsl2 설정이 깨져있는지 확인.

```PowerShell
wsl --set-default-version 2

```

확인 결과 작업 완료, 이상 없음.

Wsl/0x80070422 오류만 반복해서 나옴.

- Hyper-V 관련 서비스 의심

```PowerShell
sc.exe query vmms
```
WSL2 가 LxssManger->Hyper-V(vms)->WSL2 kernel VM 순으로 가동해서 체크
체크 결과 상태 4 RUNNING. 이상 없었다.
하이퍼바이저가 부팅에서 비활성화 되었는지 체크해 보려는데, VM은 잘 쓰고 있어서 아닐 거라 생각함.

```PowerShell
bcdedit /enum | findstr -i hypervisorlaunchtype
```
결과값 런치타입 AUTO 로 잘 세팅되어 있었다.


- 스토어 패키지 꺠짐 의심

이건 서칭중에도 발견된 문제였다. WSL이 Windows store 패키지 구조로 돌아가서 다시 설치했더니 해결되었다는 글.
언인스톨 - 인스톨 과정을 거쳤지만 오류는 해결되지 않음.

- GPT의 마지막 제안 : 실제 Linux 배포판 설치

하지만 C드라이브 용량 문제로 거절했다. 그렇게 해결될 것 같진 않았다.


### 해결: 윈도우 업데이트

결론은..
윈도우 업데이트를 최신 버전까지 다 돌렸더니 해결되었다.

발견 과정은,
WSL 재설치 과정에서 윈도우11 업데이트 알람이 떴고
거기서 WSL 리눅스 호환 패키지가 설치실패 되었음을 발견했다.
그런데 그 위에 윈도우 업데이트 패키지 보류가 떠 있던 것.
C드라이브 용량 문제로 윈도우 업데이트가 지연되고 있었고 그로 인해 발생한 호환성 문제였다.

...



### 요약
 
1. Wsl/0x80070422 에러 해결법 : 윈도우 업데이트.
2. 설치 에러가 났을 땐 OS 업데이트 상태부터 확인하자.
3. C드라이브는 항상 50GB 이상의 여유 공간을 남겨두자.


### 배운 것

WSL.exe 에 대해 알게 되었다.
WSL은 Windows Subsystem for Linux 로 윈도우에서 리눅스 커널과 사용자 공간을 실행하는 기능이다.
VM 없이 리눅스를 돌리는 기술.
VM 이 가상 컴퓨터를 생성해, 가상 자원들을 배분해 별개의 운영체제를 돌리는 것과 달리
WSL 은 윈도우 시스템 내에 리눅스를 통합하는 기술로 윈도우의 네트워크부터 자원까지 전부 공유한다.
도커를 리눅스 환경에서 돌리기 위해서 WSL 이 필요한 듯.
CLI의 wsl 키워드를 사용해서, 윈도우 내에서 리눅스 쉘 제어를 할 수 있다.
이를 통해 리눅스 커널 (WSL2) 를 호출해 UBUNTU 등 리눅스 배포판까지 사용할 수 있는 것.

윈도우 서비스 상태 확인법을 배웠다.

전체 서비스 확인 :

| services.msc

| sc query <service> 

커맨드로 STATE 가 RUNNING 인지 STOPPED 인지 확인 가능.

| sc qc <service> 

커맨드로 서비스 설정 확인. 

| sc start 
| sc stop

서비스 시작과 중지.

**windows Feature** 

| dism /online /get-featureinfo /featurename:<feature>
켜져있는지 확인
| dism /online /enable-feature /featurename:<feature> /all
기능 켜기
| dism /online /disable-feature /featurename:<feature>
기능 끄기

**hyper-V 상태 확인**

| bcdedit /enum | findstr -i hypervisor
bcd = boot configuration data.
/enum = 설정 보기.
/set = 옵션 변경.
/delete = 부팅 항목 삭제.
