## 파이썬의 특징

### 1. 모든 데이터 타입 값은 객체다

파이썬에서는 정수형 데이터 값(1, 2) 하나까지도 전부 메모리 안의 객체로 생성한다.
객체는 data + type + method 인데, 1이라는 정수형 값도 이 세 요소를 다 가지고 있다.

### 2. 거의 모든 값은 힙 영역에 존재한다.

C와 다르게 기본 데이터 타입 리터럴도 전부 힙 영역에 생성된다.
값들이 전부 객체이기 때문이고, 객체는 힙에 생성되기 때문.

### 3. 변수는 값 자체를 저장하지 않고 객체의 주소를 매핑한다.

C와 다르게 파이썬의 변수는 객체의 주소(참조)를 네임스페이스에 매핑한다.
값을 보관하는 것이 아닌 값의 메모리 주소를 연결하는 개념. 참조.
변수는 이름에 불가하다.

파이썬 변수는 다음과 같이 동작한다.

a. 코드 스코프마다 프레임 객체가 생성됨 (객체 -> 힙)
b. 프레임에는 locals, globals(모듈레벨), builtins(기본제공) 라는 네임슽페이스 dict 가 생성됨
c. 지역 변수명은 locals 네임스페이스에 key 값으로 들어감. 
  (네임스페이스도 객체 -> 힙에 생성)
d. 변수의 value 값에 변수가 매핑하는 객체의 주소값이 들어감.
e. 함수 호출 스택(stack)에서 프레임을 호출할 때, 이 힙 영역 프레임의 주소를 가져와 조작함.

### 4. 변수는 타입을 가지지 않는다.

파이썬에서 데이터 타입은 변수에 지정되는 것이 아니다.
변수에 매핑된 객체가 타입을 가진다.
그러므로 파이썬에서 변수는 타입을 가지지 않고,
런타임에 기존 객체와 다른 타입의 객체를 재바인딩할 수 있다.

### 5. mutable vs immutable

파이썬의 자료형은 값을 변경할 수 있는 자료형과 변경할 수 없는 자료형으로 나뉜다.
  immutable : int, float, str, tuple
  mutable : list, dict, set

mutable 자료형에서 값의 변경은 C처럼 슬롯에 새 값을 들여오는 것이 아니다.
모든 값이 객체이기 때문에, 슬롯은 각 객체의 주소를 참조하고 있다. 
즉 값을 변경한다는 것은 해당 위치의 참조를 다른 객체로 교체하는 것이다.
또, 값의 추가와 같이 mutable 객체 자체의 변경도 가능하다. 

### 6. 변수의 immutable 값 변경은 사실 교체다 

파이썬의 변수는 주소공간이 아니다.
a = 1을 a = 2 로 바꿀 때 메모리의 값이 바뀐다고 생각해선 안된다.
값을 변경하는 것은 사실 새 객체를 생성해 연결하는 것이다.
기존 객체는 연결된 변수가 없어질 때 GC의 대상이 된다.

### 7. interning과 재사용의 묘리

interning 이란 값이 같은 객체를 하나만 만들고 여럿이 참조해서 재사용하는 최적화 기법.
작은 정수객체, 일부 문자열, sys.intern() 으로 명시적으로 생성된 문자열 등.
파이썬의 객체 모델을 극단적으로 활용한 최적화다.
객체 캐싱이라고도 한다.

그러므로 작은 정수에 대해서는 a is b(메모리상 객체비교)를 안정적으로 사용할 수 있지만
예상치 못한 결과의 위험이 있으므로 비교는 ==으로 할 것. 
id 는 Cpython 에서 객체의 메모리주소를 반환한다.
is 는 mutable 객체의 id 비교한다만, 같은 값도 id가 다른 경우가 있으니 주의. 

### 8. reference conuter 와 GC

reference counter 의 기능은 rust의 소유권 개념을 생각하면 쉽게 이해할 수 있다.
객체를 참조하고 있는 참의 수는 객체가 사용중인지 아닌지를 파악하는 기준이 된다.
즉 reference counter 가 0이 되면 아무도 쓰지 않는 것이 분명하므로 메모리에서 즉시 해제 된다.
파이썬의 GC 는 reference counter 에서 처리하지 못하는 순환참조 (a,b가 서로참조)를 탐지, 정리한다.

### 9. 네임스페이스와 스코프 구조

네임스페이스는 객체 참조를 저장하는 dict 자료구조의 공간이다.
키 에 변수명이, 값에 객체주소가 저장되는 매핑 구조다.
사실 변수 뿐 아니라 함수명, 클래스 명 등 모든 이름이 다 매핑되어 있다.
내부 구현상으로는 C 배열처럼 동작한다고.

스코프는 변수의 탐색 범위를 말한다.
local 스코프의 변수들은 local 네임스페이스에 저장된다. 
그 판단 기준은 변수에 대입 연산이 일어나는지의 여부.
대입 연산이 없다면 LEGB 탐색을 시작한다.

그리고 네임스페이스도 객체이므로, 힙에 생성된다.
local, global, builtin 전부 객체로 따로 생성되고, 프레임은 이 네임스페이스 객체를 참조한다.

#### 변수 탐색 - LEGB 규칙과 global, non local

스코프 탐색의 순서는 Local, Enclosing, Global, Built-in 순이다.
name resolution 시에 제일 먼저 Local 네임스페이스를 확인하고, 
Local 네임스페이스에 없다면 중첩 함수의 바깥 함수 스코프 Local변수인지 확인한다.  (Enclosing)
상위 함수가 없다면 Global 네임스페이스를 체크.
마지막으로 Builtin 네임스페이스를 체크한다.

변수에 golbal이 선언되어 있다면 이름을 global 네임스페이스에 바인딩.
nonlocal 이 설정되었다면 enclosing 스코프 변수를 수정.

### 10. 프레임 객체와 실행 컨텍스트

프레임이란 함수,모듈 등의 실행 컨텍스트(문맥)다.
그 내부에는 local/global/builtin namespace,
코드 정보(객체),
이전 프레임 링크가 있다. (trace,디버깅,체인 탐색을 위함)

파이썬은 소스코드를 컴파일한 바이트코드로 컴파일해 코드 객체에 넣는다.
그 바이트코드가 프레임에서 instruction pointer 를 통해 순차실행된다.
실행 중에 새 함수가 호출되면, 그 함수의 프레임이 스택의 최상단에 생성된다.
함수 실행이 종료되면 프레임은 pop되고, 이전 프레임이 계속 실행된다.

### 11. call stack과 함수 호출 (call stack)

파이썬의 call stack은 객체가 아니다. 인터프리터가 관리하는 실제 구조다.
프레임들은 call stack 에 스택으로 쌓여 있고, (LIFO)
동시에 f_back으로 서로 링크되어 있다.(연결리스트)

함수를 호출한다는 것은 새 프레임을 생성하고 그것을 call stack에 push 하는 것.

(프레임 생성 : local namespace 생성, global, builtin 참조 연결
 -> 인자 바인딩 (이름들을 namespace에 등록) -> call stack 에 push
 -> Inst-Pointer 를 함수코드 시작 위치로 이동 -> 바이트코드 실행)

즉 실행 흐름을 만들어서 스택으로 이동시키는 것이다.

### 12. 참조 증가/감소 메커니즘

참조는 곧 객체의 생명주기다.
참조되지 않는 객체는 메모리에서 즉시 해제된다.
참조 카운트는 객체에 적용되기에 함수나 프레임같은 모든 객체가 ref count를 가짐.

참조 카운트는 변수 바인딩시, 컨테이너 내부 객체 저장시, 함수 호출시,
리턴값을 다른 네임이 받을 시 등 증가한다.
프레임은 제거시 참조가 감소한다.

### 13. 함수 호출 시 call by object reference

call by value 도 call by reference 도 아니다.
함수의 인자가 객체 참조를 전달받는다.
그래서 인자는 함수 외부의 변수와 같은 객체를 가리킨다.

mutable 객체의 경우 내부 인자로 변경시 같은 객체를 참조하고 있으므로 외부 변수도 같이 변경
immutable 객체는 변경시 재바인딩되므로 내부 인자만 재바인딩되고 외부는 그대로. (변수 따라감)

### 14. mutable 객체의 공유와 사이드 이펙트

mutable 객체는 여러 변수가 공유(참조) 가능
그래서 함수에 있는 인자가 외부 변수 상태를 바꿔버림
.copy() 복사를 사용하라

### 15. 얕은 복사 vs 깊은 복사

shallow copy : 겉만 복사해서 새로 객체 하나 만듬. 내부의 객체는 공유함. 바로가기. (side effect)
deep copy : 내부 요소들 전부 다 새 객체로 복사함. 별도의 객체. 복사본.

슬라이싱은 얕은 복사.

### 16. 모든 object의 원형은 PyObject 

모든 object는 PyObject 를 기반으로 생성. int,list,str,함수,클래스 전부.
그래서 다 같은 방식으로 처리됨. PyObject 구조기반이니까.
PyObject 는 ob_refcnt (레퍼런스카운트) 와 ob_type(데이터타입) 을 가짐.
객체 = 데이터 + 타입 + 레퍼런스카운트.

파이썬 다형성의 핵심. 동적 타이핑의 원리. 메모리 관리의 기반.
파이썬 객체 시스템의 중추, 핵심 개념.

### 17. 리터럴과 객체 생성 타이밍

리터럴은 객체 생성 요청하는 코드. (재사용 제외)
immutable 객체는 컴파일 타임에 코드 객체의 상수 영역으로 들어간다 (정수,문자열 등)
mutable 객체는 런타임에서 실행할(프레임이 생성될) 때마다 새 객체를 생성.
상황에 때라 재사용 가능.

코드객체는 프레임의 f_code 가 참조하고 있음.
바이트 코드, 상수 영역, 변수 이름(값이 없는 단순 문자) 등 가짐.
프레임이 생성될 때 네임스페이스는 코드 객체를 보고 어떤 이름이 필요한지 확인함.


### 18. 코드 객체 (code object)

컴파일 시점에 생성되는 바이트코드. immutable, 객체임.
프레임 생성시에 프레임이 f_code 로 참조하고, Inst-pointer 로 실행함.
eval, exec 등은 런타임에 코드 컴파일이나 코드 객체 생성 가능.

동일 함수는 동일 코드 객체 참조. 

### 19. 함수 객체 (function object)

함수 선언시에 생성되는 함수의 객체. 
함수 객체는 함수의 정의 자체를 표현. 코드 객체를 참조함.
함수객체가 함수를 정의했다면 프레임은 매 실행시마다 생성된다고 보면 됨.
프레임은 함수 객체 참조 안함. 함수 실행에는 코드객체와 네임스페이스만 필요하기 때문.


### 20. 상수 풀 (constant pool)

코드 객체 내에 있는 imuutable 객체를 재사용하기 위한 영역
code 객체의 co_consts 에 존재
같은 코드 블록 안에서만 공유돼서 global, 함수 등은 상수 풀 다 따로 가짐.
interning 과는 살짝 다름. 상수 풀은 코드 단위, 인터닝은 전역적/최적화 <<이부분 헷갈리네
